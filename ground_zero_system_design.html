1. Entity Model — The Four Actors + Two AI Agents

Ground Zero has four human entities and two AI entities. Each entity both produces signals that feed the learning graph and consumes outputs from it. The system is designed so that every entity interaction generates at least one learnable signal.

🎓

Student

The central entity. Every signal ultimately describes or updates a student's learning state.

Produces

  • MCQ/short answer responses
  • Voice diagnostic transcript
  • Session speech (via transcript)
  • SPARK chat conversations
  • Confidence self-reports
  • Response timing data
  • Project artifacts

Consumes

  • Adaptive question selection
  • AI tutor hints/conversations
  • Skill map progress visualization
  • Personalized review schedule
👤

Teacher / Facilitator

Runs live sessions. Minimally annotates. Primary role is facilitating, not data entry.

Produces

  • 3-tap engagement rating per student
  • Notable moment free-text (optional)
  • Intervention flags
  • Question push (live activity)

Consumes

  • Live class dashboard (BKT states)
  • Stuck student alerts
  • Suggested next questions
  • Session analytics & class trends
👨‍👩‍👧

Parent

Passive consumer. Never produces learning signals. Receives weekly AI-generated narrative reports.

Produces

  • Nothing (by design)

Consumes

  • Weekly narrative report (AI-generated)
  • Mastery progress across 4 pillars
  • Curiosity interest map from SPARK
  • Suggested home activities
  • Intervention alerts (from facilitator)
🤖

SPARK — AI Companion

Student-facing conversational AI. Free-range exploration. Richest curiosity signal. Also delivers onboarding diagnostic.

Produces

  • Chat transcripts (every turn logged)
  • Onboarding diagnostic results (P(L₀) seeds)
  • Topic classification per conversation
  • Prompt quality scores

Consumes

  • Student's current learning graph state
  • Curriculum context for guided mode
  • Stuck detection triggers from ABM
🧠

ABM Engine — Learning Graph Brain

The Adaptive Bayesian Mastery engine. Processes all signals, maintains per-student-per-skill probability states across all 4 pillars.

Produces

  • P(L) updates per skill node per student
  • Stuck detection alerts
  • Next question recommendation
  • Mastery predictions & review schedule
  • Weekly report data for parent surface

Consumes

  • All 7 signal sources (below)
  • Skill graph topology (prerequisites)
  • BKT parameter priors per skill

3. Signal Ingestion Pipeline — 7 Sources into the Learning Graph

Each signal source has a different latency, format, and processing requirement. The key design decision: everything is event-driven. Each raw signal produces an event on the bus. Processors subscribe to events and produce learning graph updates.

#Signal SourceRaw FormatLatencyProcessingGraph UpdatePillars Fed
1 MCQ MCQ Response { questionId, correct, responseTimeMs, attempts } Real-time
(~100ms)		 Direct BKT Bayes update + response time weighting P(L) update on skill node. Strongest immediate signal. Math AI/Sys
2 CONFIDENCE Confidence Self-Report { questionId, level: got_it|kinda|lost|too_fast } Real-time
(with MCQ)		 Confidence calibration modifier on BKT update. Overconfidence/underconfidence detection. Modifies P(L) update weight. Adjusts personal P(G), P(S) over time. Math AI/Sys
3 VOICE DIAG Voice/SPARK Diagnostic Conversation transcript + AI analysis per skill Pre-course
(~20 min)		 LLM extracts: conceptual depth, confidence bias, response latency, signals per skill Seeds P(L₀), P(G), P(S), P(T) per student per pillar. Cold-start killer. Comm Create AI/Sys Math
4 TRANSCRIPT Session Transcript Speaker-diarised text log of full live session Post-session
(~5 min processing)		 NLP pipeline: talk time, reasoning connectives, peer-building, position changes, vocabulary sophistication, question detection Batch P(L) updates across Communication + Creativity nodes. Longitudinal trend data. Comm Create Math
5 SPARK SPARK Chat Log Full student-AI conversation (turns + timestamps) Near real-time
(per session)		 NLP: topic modelling, depth scoring, hypothetical detection, pushback detection, prompt iteration, use mode classification Curiosity interest map. Creativity/Curiosity pillar updates. AI literacy signals for Pillar 3. Create AI/Sys Comm
6 FACILITATOR Facilitator Notes { studentId, engagement: 1|2|3, notableMoment?, interventionFlag } Post-session
(~2 min input)		 Direct mapping: engagement → learning rate modifier. Intervention → parent alert queue. Engagement quality modifies session-level P(T). Intervention flags bypass learning graph. All pillars (engagement is cross-cutting)
7 ARTIFACT Project Artifacts Product cards, pitch decks, system maps, final Demo Day project Periodic
(per level)		 Rubric evaluation (facilitator + AI joint scoring). NLP on written content. Semantic comparison across versions. Capstone updates to Builder Mindset, Creativity, Communication nodes. Highest-weight signals. AI/Sys Create Comm Math
The multi-source fusion principle: A single skill node (e.g., "Logical Argumentation") can receive evidence from MCQ (structured test), Transcript (spoken reasoning), SPARK (pushback on AI), and Facilitator (engagement flag). Each source has a reliability weight: MCQ = 1.0 (most precise), Transcript NLP = 0.7 (noisy but rich), SPARK = 0.6 (self-directed, less controlled), Facilitator = 0.5 (subjective). The ABM engine applies Bayesian updates from each source with these weights, so no single source dominates.

4.2 Per-Student State (what ABM tracks for each student × each skill node)

// Core BKT state — exists for every (studentId, skillNodeId) pair
interface SkillMasteryState {
  studentId:        string;
  skillNodeId:      string;
  pillar:           "communication" | "creativity" | "ai_systems" | "math_logic";

  // BKT Core
  pLearned:         number;      // P(L) — current mastery probability [0, 1]
  attempts:         number;      // total evidence events received
  correct:          number;      // total positive evidence events

  // Individualized Parameters (from voice diagnostic + online learning)
  personalPG:       number;      // per-student guess rate
  personalPS:       number;      // per-student slip rate
  personalPT:       number;      // per-student learning rate for this skill
  confidenceBias:   number;      // [-1, 1] overconfident ↔ underconfident

  // Temporal
  lastEvidenceAt:   number;      // timestamp of last signal
  stability:        number;      // forgetting curve half-life (days)
  nextReviewDue:    Date;        // optimal review scheduling (HLR)

  // Stuck Detection
  consecutiveFails: number;
  isStuck:          boolean;
  stuckTriggeredAt: Date | null;

  // Evidence Sources — which sources have contributed to this node
  evidenceSources: {
    mcq:          number;      // count of MCQ events
    transcript:   number;      // count of transcript-derived events
    spark:        number;      // count of SPARK-derived events
    facilitator:  number;      // count of facilitator events
    artifact:     number;      // count of artifact events
    voiceDiag:    number;      // count of voice diagnostic events (0 or 1)
  };

  // Stage (derived from P(L) thresholds per pillar)
  currentStage:     1 | 2 | 3 | 4 | 5;
  masteredAt:       Date | null;
}

4.3 Multi-Source Evidence Fusion

When the ABM engine receives an evidence event, it applies the Bayesian update with a source-specific reliability weight:

SourceWeightRationaleExample Event
MCQ1.0Binary correct/incorrect. Most precise signal. Direct BKT input.{ correct: true, responseTime: 4200ms }
CONFIDENCEModifierNot a separate update — modifies the MCQ update weight (0.6x to 1.35x){ level: "got_it", outcome: "wrong" } → 1.35x negative
VOICE DIAGInit onlySeeds parameters. Doesn't produce ongoing updates.{ pL0: 0.35, pG: 0.30, confidenceBias: 0.4 }
TRANSCRIPT0.7NLP-derived. Rich but noisy. Batch-processed post-session.{ reasoningConnectives: 4.2/100words, peerBuilding: true }
SPARK0.6Self-directed. High variance. Best for curiosity/creativity signals.{ topicDepth: 12turns, hypotheticalFraming: true }
FACILITATOR0.5Subjective human judgment. Valuable for what algorithms can't see.{ engagement: "high", notableMoment: "..." }
ARTIFACT0.9Capstone evaluation. Infrequent but high-weight. Joint AI+human scoring.{ rubricScore: 8/10, feedbackLoops: true }

4.4 Cross-Pillar Skill Transfer

Some skills span multiple pillars. When a student demonstrates "Causal Reasoning" in a session transcript (Communication context), the ABM engine also applies a transfer update to the "Correlation vs Causation" node in Math/Logic — at a reduced weight (0.3x). This is modelled as a cross-pillar correlation matrix:

5.1 Student Entity

interface Student {
  id:               string;        // UUID
  name:             string;
  grade:            4 | 5 | 6 | 7 | 8 | 9;
  gradeBand:        "4-5" | "6-7" | "8-9";
  cohortId:         string;
  enrolledAt:       Date;

  // Learning Graph — the core state
  skillStates:      Map<string, SkillMasteryState>;  // skillNodeId → state
  pillarStages:     {
    communication:  1 | 2 | 3 | 4 | 5;
    creativity:     1 | 2 | 3 | 4 | 5;
    aiSystems:      1 | 2 | 3 | 4 | 5;
    mathLogic:      1 | 2 | 3 | 4 | 5;
  };

  // Voice Diagnostic Profile (set once, pre-course)
  diagnosticProfile: {
    completedAt:     Date;
    confidenceBias:  number;    // global: overconfident (+) or underconfident (-)
    responseLatency: "fast" | "medium" | "slow";
    conceptualDepth: "surface" | "procedural" | "conceptual";
    interestAreas:   string[];  // topics from SPARK onboarding
  } | null;

  // Curiosity Profile (built from SPARK over time)
  curiosityProfile: {
    topInterests:    { topic: string; sessionCount: number; avgDepth: number }[];
    sparkUseMode:    "understanding" | "answer_getting" | "mixed";
    totalSparkTurns: number;
    pushbackRate:    number;   // % of sessions with AI pushback
  };

  // Engagement Metrics
  sessionCount:      number;
  totalPracticeMin:  number;
  streakDays:        number;
  lastActiveAt:      Date;
}

5.3 Evidence Event (the universal signal format)

// Every signal source produces events in this format.
// This is what flows through the event bus.
interface EvidenceEvent {
  id:               string;        // UUID
  studentId:        string;
  skillNodeId:      string;        // target skill node in learning graph
  pillar:           Pillar;
  timestamp:        Date;

  // Source identification
  source:           "mcq" | "transcript" | "spark" | "facilitator" | "artifact" | "voice_diagnostic";
  sourceWeight:     number;        // reliability weight (0.5–1.0)
  sessionId:        string | null;

  // Evidence payload (varies by source)
  payload: {
    // MCQ events
    correct?:       boolean;
    responseTimeMs?:number;
    confidence?:    ConfidenceLevel;
    attempts?:      number;

    // Transcript NLP events
    connectiveRate?:  number;    // per 100 words
    elaborationDepth?:number;    // avg words per turn
    peerBuilding?:    boolean;
    positionChanged?: boolean;
    withJustification?:boolean;
    novelContribution?:boolean;

    // SPARK events
    topicDepthTurns?:  number;
    hypotheticalFrame?:boolean;
    pushedBackOnAI?:   boolean;
    promptIteration?:  boolean;
    useMode?:         "understanding" | "answer_getting";

    // Facilitator events
    engagement?:      "engaged" | "partial" | "motions";
    notableMoment?:   string;

    // Artifact events
    rubricScore?:     number;    // 0-10
    feedbackLoops?:   boolean;
    ideaEvolution?:   number;   // semantic distance from v1

    // Voice diagnostic events
    estimatedPL0?:    number;
    estimatedPG?:     number;
    estimatedPS?:     number;
    estimatedPT?:     number;
    confidenceBias?:  number;
  };
}

6.2 Processing Latency Tiers

TierLatencyEventsProcessing
Hot Path < 200ms MCQ response + confidence Direct BKT update → immediate P(L) change → next question selection. Must not block the student.
Warm Path 1–30 seconds SPARK chat turn, facilitator note Near-real-time processing. SPARK topic classification, engagement modifier application.
Cold Path 1–10 minutes Session transcript, project artifact Batch NLP pipeline. Speaker diarisation, connective extraction, originality scoring. Produces multiple EvidenceEvents.
Scheduled Daily / Weekly Forgetting decay, parent reports Cron job applies temporal decay to all P(L) states. Weekly report generation from graph snapshots.

7. Consumer Surfaces — Who Sees What

Student App

Live Class View: Video (100ms) left, Live Activity (questions) right, confidence bar ("How are you feeling?") below, AI tutor chat at bottom.

Practice View: Adaptive question flow driven by ABM scheduler. Shows mastery progress per skill.

Skill Map: 4-pillar spider/web visualization. Node states: locked → ready → learning → mastered. Tap to practice.

SPARK: Free-range AI chat. Conversation topics inform curiosity profile.

Teacher Dashboard

Live Class: Real-time grid of student P(L) states. Stuck alerts bubble up. Suggested questions based on class-wide gaps. Push questions to all students.

Post-Session: 3-tap annotation form (engagement, notable moment, intervention flag). Session analytics: talk time distribution, question accuracy heatmap.

Class Trends: Cohort-level mastery progression across 4 pillars over time. Identify students who need attention.

Parent Report (Weekly)

AI-generated narrative: "This week, Aarav showed strong growth in Logical Argumentation — he used 3x more reasoning connectives in discussion than last week. He's still building confidence in Fractions; the AI tutor helped him through a stuck moment on Wednesday."

4-Pillar Progress Web: Visual spider chart showing stage progression per pillar.

Curiosity Map: "Aarav spent 45 minutes exploring space exploration with SPARK this week. He asked 12 follow-up questions."

Home Activities: AI-suggested activities based on current learning gaps.

Admin Analytics

Cohort Metrics: Average P(L) per pillar per cohort. Questions-to-mastery trends. Stuck rate by skill node.

Signal Health: Event pipeline throughput. NLP processing latency. Source coverage per student.

Model Accuracy: AUC tracking for BKT predictions. Confidence calibration accuracy per cohort.

The one-sentence summary: Every student interaction — answering a question, speaking in class, chatting with SPARK, building a project — produces an evidence event that flows through an event bus, gets processed by source-specific analyzers, and updates a per-student Bayesian learning graph across 4 pillars and ~152 skill nodes. Teachers see real-time states. Parents see weekly narratives. The AI gets smarter with every event. The dataset that accumulates is the moat.
