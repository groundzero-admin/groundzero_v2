Key Design Principles

Curriculum-Agnostic Engine

The mastery engine knows nothing about specific curricula. It processes evidence events and maintains competency states. Any curriculum module registers its activities with competency mappings.

Model Swappability

The mastery engine exposes a stable interface. BKT sits behind it today. DKT, transformers, or ensembles replace it tomorrow without changing anything above or below.

LLM-as-Assessor

Communication and Creativity cannot be measured by MCQ alone. Transcripts and SPARK chats are assessed by an LLM, producing structured evidence events that feed the same engine as MCQ responses.

Graph as Data, Not Code

The competency model, prerequisite edges, and co-development weights are stored as data — versionable, A/B testable, and expandable without code changes.

3. Skill Graph

The skill graph defines how competencies relate to each other. Two types of edges: prerequisite (must master before progressing) and co-development (evidence for one provides partial evidence for the other).

4. Mastery Engine

The mastery engine is the brain of the learning graph. It processes evidence events, maintains per-student per-competency mastery states, and exposes a stable interface that never changes — even when the underlying model is swapped.

4.1 Interface Contract

Any model that implements this interface is a valid mastery model:

MasteryEngine {
  // Student lifecycle
  initStudent(studentId, diagnosticProfile) → StudentState

  // Core: process evidence, return updated state
  processEvidence(studentId, evidence: EvidenceEvent) → StudentState

  // Read state
  getStudentState(studentId) → StudentState
  getCompetencyState(studentId, competencyId) → CompetencyState
  getCapabilityState(studentId, capabilityId) → CapabilityState

  // Predictions
  predictPerformance(studentId, competencyId) → { probability, confidence }

  // Temporal
  applyDecay(studentId) → StudentState

  // Graph
  getPrerequisites(competencyId) → CompetencyId[]
  getReadyCompetencies(studentId) → CompetencyId[]
}

4.2 BKT Node State

BKTNodeState {
  pLearned:          number    // P(L) — current mastery [0, 1]
  pGuess:            number    // P(G) — personal guess rate
  pSlip:             number    // P(S) — personal slip rate
  pTransit:          number    // P(T) — personal learning rate

  totalEvidence:     number    // count of all evidence events
  positiveEvidence:  number    // count of positive outcomes
  consecutiveFails:  number
  isStuck:           boolean

  lastEvidenceAt:    Date
  stability:         number    // forgetting half-life (days)

  // Derived
  stage:             1 | 2 | 3 | 4 | 5
  trend:             "improving" | "stable" | "declining"
  confidence:        number    // how reliable is this P(L) estimate
}

4.3 The BKT Update Algorithm (10 Steps)

1

DECAY — Apply forgetting since last evidence

daysSince = (now - lastEvidenceAt) / days
pL = pL0 + (pL - pL0) × e^(-daysSince / stability)

Knowledge decays exponentially toward the prior. High stability = slower decay.

2

OBSERVE — Bayesian posterior update

// Positive evidence (outcome ≥ 0.5):
pCorrectGivenL = (1 - pSlip)
pCorrectGivenNotL = pGuess
pL_posterior = (pL × pCorrectGivenL) /
               (pL × pCorrectGivenL + (1-pL) × pCorrectGivenNotL)

// Negative evidence (outcome < 0.5):
pWrongGivenL = pSlip
pWrongGivenNotL = (1 - pGuess)
pL_posterior = (pL × pWrongGivenL) /
               (pL × pWrongGivenL + (1-pL) × pWrongGivenNotL)
3

WEIGHT — Source reliability + response time

pL_weighted = pL + (pL_posterior - pL) × event.weight × speedModifier
Speed × OutcomeModifierMeaning
Fast + Correct1.3Confident mastery
Slow + Correct0.6Effortful, maybe guessing
Fast + Wrong0.7Careless slip
Slow + Wrong1.2Genuine gap
4

LEARN — Learning transition P(T)

pT_adjusted = pTransit
if aiInteraction == "hint":         pT × 1.5
if aiInteraction == "conversation": pT × 2.0

pL_new = pL_weighted + (1 - pL_weighted) × pT_adjusted

AI-assisted learning accelerates the transition.

5

CONFIDENCE MODIFIER

if confident + wrong:      amplify negative × 1.3
if unconfident + right:    dampen positive × 0.8
Update personal confidenceBias over time
6

STUCK DETECTION

if outcome < 0.5: consecutiveFails++
else: consecutiveFails = 0
isStuck = consecutiveFails ≥ 4

Triggers teacher dashboard alert and possible SPARK intervention.

7

STABILITY UPDATE

if positive evidence AND pL > 0.7:
  stability = min(60, stability × 1.4)

Successfully demonstrated mastery increases forgetting half-life.

8

CO-DEVELOPMENT PROPAGATION

for each co-development edge from this competency:
  Generate derived EvidenceEvent for linked competency
  weight = original_weight × transfer_weight
  Process through same pipeline (max 1 hop)
9

DERIVE STAGE

stage = f(pL)   // Stage 1-5 threshold function
trend = compare pL to pL from 5 evidence events ago
10

DERIVE CONFIDENCE

confidence = f(totalEvidence, recency, variance)
// More evidence + recent + low variance = high confidence

5. Evidence Pipeline

Every signal source produces evidence events in a universal format. This is what flows through the processing queue and feeds the mastery engine.

5.1 EvidenceEvent Schema

interface EvidenceEvent {
  id:            string         // unique event ID
  studentId:     string
  competencyId:  string         // target competency (C1.1 through C4.19)
  timestamp:     Date

  // Source
  source:        "mcq" | "llm_transcript" | "llm_spark" |
                 "facilitator" | "artifact" | "diagnostic"
  moduleId:      string         // "level_1" | "math_v1" | "spark"
  sessionId:     string | null

  // The observation
  outcome:       number         // 0-1 (binary for MCQ, continuous for LLM)
  weight:        number         // source reliability (0.5-1.0)

  // Metadata
  meta: {
    responseTimeMs?:  number
    confidence?:      "got_it" | "kinda" | "lost"
    attempts?:        number
    aiInteraction?:   "none" | "hint" | "conversation"
    evidenceText?:    string    // actual quote (for audit)
  }
}

6. SPARK Diagnostic

A 10-minute conversational assessment that seeds the learning graph before a student’s first class. Without it, every student starts at P(L) = 0.10 for all 56 competencies — the first 3–4 sessions would be wasted on calibration.

6.1 Flow

1Student opens SPARK for the first time
2SPARK introduces itself as friendly AI companion
3~15-20 adaptive questions over ~10 minutes
4DiagnosticProfile generated, graph seeded
5Seamless transition to companion mode

6.2 Adaptive Question Selection

Information-theoretic approach: At each step, SPARK picks the question that maximally reduces uncertainty across unassessed capabilities.

for each candidate question Q:
  expectedInfoGain(Q) = Σ over capabilities C that Q assesses:
    H(C_current) - E[H(C_after_answer)]

  where H(C) = entropy of current stage estimate

Pick Q with highest expectedInfoGain

Early questions target multiple capabilities (high info gain per question). Later questions target capabilities with the most remaining uncertainty. If a student is clearly strong in one area, SPARK doesn’t waste time confirming.

For each competency under a capability: (1) Start with capability stage → P(L₀). (2) Apply competency override if one exists. (3) Ensure prerequisite consistency — bump prerequisites up if a downstream competency is estimated higher. (4) Set personal BKT parameters from diagnostic. Within 3–4 sessions, real evidence starts overriding the diagnostic priors. The diagnostic just ensures the student isn’t bored (everything too easy) or lost (everything too hard) during those first sessions.

7. Curriculum Predictor

Given a student’s state across 56 competencies + a registry of available activities, what should they do next? The predictor evolves from rule-based (MVP) to contextual bandit (month 3–6) to RL path planner (month 12+).

7.1 Activity Registry

Activity {
  id:             string
  moduleId:       string          // "level_1" | "math_v1" | ...
  name:           string
  type:           "warmup" | "key_topic" | "diy" | "ai_lab" | "artifact"

  primaryCompetencies:   { competencyId, expectedGain }[]
  secondaryCompetencies: { competencyId, expectedGain }[]
  prerequisites:         { competencyId, minStage }[]

  duration:       number          // minutes
  gradeBands:     ("4-5" | "6-7" | "8-9")[]
  signalSources:  ("mcq" | "short_answer" | "transcript" | "spark" | "artifact")[]
}

7.2 Phase 1 — Rule-Based (MVP, Day 1)

Scoring Function

Score(activity, student) =
  Σ (competency_gap × expected_gain × prerequisite_met)

Where:
  competency_gap  = 1.0 - P(L)  // bigger gap = more room to grow
  expected_gain   = activity's expected improvement for this competency
  prerequisite_met = 1 if all prereqs ≥ Stage 2, else 0

Expert curriculum as default. The 14-session Level 1 sequence is the default path. The predictor makes deviations: if SPARK diagnostic shows Stage 3+ on a competency that Session 2 targets → skip or enrich. If Stage 1 on a prerequisite → scaffold first.

Within-activity adaptation: For MCQ activities, select questions in the student’s zone of proximal development (P(L) 0.3–0.7). Too easy (>0.85): skip. Too hard (prereqs not met): scaffold.

7.3 Phase 2 — Contextual Bandit (Month 3–6)

Bandit Formulation

Context = student state vector (56-dim P(L) + metadata)
Arms    = available activities
Reward  = learning gain (P(L) delta after activity)

Training data: every MVP session produces (student_state_before, activity, student_state_after). Thompson sampling or epsilon-greedy for exploration.

7.4 Phase 3 — RL Path Planner (Month 12+)

RL Formulation

State   = student competency vector + session history
Action  = next activity
Reward  = mastery gain + engagement + Demo Day readiness

Plans multi-session sequences. Optimizes long-term outcomes with a compound objective.

Curiosity Map: Topics explored in SPARK this week.

Home Activities: “Your child showed interest in X — try Y at home.”

Admin Analytics

Cohort Metrics: Aggregate competency distributions, progress rates, stuck rates.

Signal Health: Are all 7 sources producing data? Latency monitoring. LLM quality checks.
